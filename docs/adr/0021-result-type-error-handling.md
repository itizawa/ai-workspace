# ADR-0021: Result 型によるエラーハンドリング（想定内エラーの値化）

- ステータス: Accepted
- 日付: 2026-06-10
- 関連 Issue: #291

## コンテキスト（背景）

現状、ユースケース・永続化層のエラーは `common` の `AppError` 階層（`NotFoundError` / `ConflictError` / `ForbiddenError` 等）を `throw new ...` / `next(new ...)` で伝播させ、Express の集約 `errorHandler`（`server/src/middleware/errorHandler.ts`）が HTTP レスポンスに写像している。

この設計には以下の問題がある:

1. **制御フローが例外依存**: 想定内のドメインエラー（リソース未発見・権限不足等）が例外で表現され、呼び出し元が try/catch を強いられる
2. **型安全性の欠如**: 関数のシグネチャからエラーの種類が読み取れない
3. **純粋性との乖離**: ADR-0005 で定めた「関数型志向の純粋ロジック」と噛み合わない

`Result<T, E>` 型（失敗を値として）に移行すれば型でエラー処理を強制でき、想定内エラーと想定外例外を明確に分離できる。

## 決定

**(a) Result 型の採用**: 自前の軽量 Result 型を `common` に実装する（外部ライブラリ不採用）。

**(b) 想定内 / 想定外の境界**:

- **想定内エラー（Result で表現）**: ドメインルールの違反（NotFound / Conflict / Forbidden / BadRequest 相当）
- **想定外例外（throw のまま）**: プログラミングエラー・インフラ障害（DB 接続失敗・環境変数未設定等）。Express の最終ハンドラ `errorHandler` が処理する

**(c) common への配置**: `common/src/result/` に `Result<T, E>` 型と `DomainError` 型を置く（ADR-0005 整合）

**(d) AppError 階層との共存**: 既存の `AppError` 階層は削除しない。ミドルウェア層（`requireAuth` / `requireAdmin`）・インフラ層の throw 経路は変更しない。新規のユースケース経路から段階的に Result へ移行する

## 理由

### なぜ自前実装か（neverthrow / fp-ts 不採用の理由）

- `common` の外部依存を最小限に保つ（ADR-0005）
- TypeScript の判別共用体で `Result<T, E>` は十分表現できる
- `neverthrow` は Result チェーン API（`map`, `andThen` 等）を提供するが、現在のユースケースは単純な早期 return パターンで十分
- `fp-ts` は関数型プログラミング全般のライブラリで本プロジェクトのスコープに対して過剰

### なぜ段階移行か

- 既存の `throw` 経路は 20+ 箇所あり、一括移行はリグレッションリスクが高い
- 「全 throw の機械的撲滅」は目的ではなく、型安全性の向上が目的
- ミドルウェア・インフラ層の throw は ADR-0004（Express エラーハンドラ）の責務として適切に機能している

## 検討した代替案

- **neverthrow**: 型安全な Result チェーン API を提供。外部依存追加と `common` の依存増加が懸念。将来チェーン API が必要になった場合に再評価する
- **fp-ts**: 強力な関数型ライブラリ。学習コスト・過剰な依存追加で MVP 規模に対して不適切。不採用
- **一括移行**: 全 throw を一度に Result 化。リグレッションリスク大。段階移行を採用
- **現状維持**: 型安全性の問題が残る。採用しない

## 影響（結果）

- **良い影響**:
  - ユースケース関数のシグネチャからエラー種別が読み取れる
  - 呼び出し元が Result チェックを型で強制される（コンパイル時検出）
  - 想定内 / 想定外の境界が明確になり、ErrorHandler の責務が限定される
- **トレードオフ / 注意点**:
  - AppError 階層と DomainError 型が並存する（移行期の二重管理）
  - ルート層に Result → HTTP 写像の定型コードが増える
- **フォローアップが必要なこと**:
  - 全ユースケースへの段階展開（#291 では代表経路に絞る）
  - AppError 階層の廃止（全移行完了後）

## #288・ADR-0012 との関係

- **#288**（関数ファクトリ化）: 同じ関数型志向の方向性。本 ADR と方針を揃え、並行可能
- **ADR-0012**（手動 DI）: DI パターンに変更なし。Result を返す関数も同じ手動 DI で注入する
