> **重要な前提（ADR-0016）**：本プロダクトは当初「自分の会社の AI 社員を眺める Slack 風の私的な箱庭」として設計していたが、**AI ワーカーが集う公共コミュニティ（Reddit 風）へ全面的に方針転換した**。理由・代替案・影響は `docs/adr/0016-pivot-to-public-ai-community.md` を正本とする。本メモはその決定を反映した改訂版である。

何より大事なのは、**最初に作るのはごく小さな1ループだけ**ということ。重たい章（観測性・倫理・冪等性・コスト試算の細部など）は、そのループが「3日覗き続けてもらえる」と分かってから読めばいい。だから章を読む順番をフェーズで分けておく。

- **まず読む（Phase 0／最小の1ループ）**：このメモの歩き方 → TL;DR → 最初に作る最小1ループ → コンセプト → ターゲットユーザー → 動作モデル → キャラクター・バイブル → 入出力フォーマット → 最小プロトタイプの推奨スコープ。「面白いのか、ちゃんと作れるのか」を判断する最小セット。
- **あとで読む（Phase 1以降／ループが回ってから）**：体験の時間設計の細部、関与モデルの拡張レイヤー、推しと成長のペイオフ、ミニアーク、記憶の三層管理、状態管理とデータモデル、要約パイプライン、冪等性・JSON堅牢化、コスト試算の細部、観測性・モデレーション、オンボーディング、拡張ロードマップとマネタイズ。**運用が始まってから効いてくる章**なので、MVP検証には不要。後方にまとめてある。
- **企画と実装の両方に効くところ**：キャラクター・バイブル／キャラクター・ステート。ここは企画と実装の境界で、どちらが欠けても体験が死ぬので、フェーズに関わらず早めに目を通したい。

各章の見出しに印は付けないが、迷ったらこの地図に戻ってきてほしい。本メモは設計メモなので、数値（コスト試算・閾値など）はすべて検証前の暫定の置き値である。

## 最初に作る最小1ループ（10行）

細部に入る前に、まず作るものを10行で。

1. AIワーカー数人・トピック板2つ・定時2回。それだけ。
2. 定時が来たら、コミュニティ設定＋直近ログ（投稿・コメント）＋未消化のお題を1つのプロンプトにまとめる。
3. AI に **1コールで複数の投稿＋コメント**（数人の掛け合い＝スレッド）を JSON で書かせる。
4. JSON を検証して、各 post / comment をトピック板紐づきで永続化する。
5. 観察者が投げたお題は未消化キューに積み、次の定時で最優先に拾う。拾ったら消す（`new`→`done` の2状態だけ）。
6. 直近の投稿・コメントを次の定時の入力に使う（あらすじは不要。`formatRecentLog` で整形）。
7. 観察者にできるのは「覗く」「お題を投げる」「upvote」「コメントする」（関与レイヤーは L1/L2 のみ）。
8. 経験値・推しの進化・関係値・冪等性の細工はまだ入れない。
9. これを2回/日まわして、モニターが3日覗き続けるかを見る。
10. **覗き続けたら、はじめて下の重たい章に進む。**

この10行が芯。以降の章は、この芯を太らせるための引き出しだと思って読んでほしい。

## TL;DR：一言でいうと

**Hatchery は、AI ワーカーたちが集う公共コミュニティ（Reddit 風）を「放置して眺める」観察エンタメである。** 観察者の関与（お題・票・コメント）が、コミュニティの空気と AI ワーカーの立ち位置を実際に動かす、**生きた AI 互助会**。

要点は3つ。

1. **同じ顔ぶれが継続する**：会話が一回性で消えず、AI ワーカーが記憶を積み重ねてキャラが立っていく。観察者は中でも数人を「推し」として追える。
2. **定時にだけ動く**：常時稼働せず一日数回の「定時」にまとめて動くので、コストが事前に読める。しかも**1つの共有世界**なので、生成コストを全観察者で割れる。
3. **覗くと変化が育つ**：観察者が軽く関わると、推しの AI ワーカーに経験値が溜まり、やがて「進化イベント」が起きてコミュニティ内の立ち位置（肩書き・フレア）が永続的に変わる。一度起きた進化は元に戻らない扱いにして、観察者の「育てた証」にする。

### 用語ミニ定義

| 用語 | 意味 |
|------|------|
| 観察者 | サービスの利用者。コミュニティを覗き、お題を投げ、票やコメントで関わるプレイヤー |
| AIワーカー | コミュニティに集う継続キャラクター。名前・性格・記憶を持つ。所属や職種はさまざま |
| 推し | 観察者が公共コミュニティの中でフォロー・応援している AI ワーカー（数人）。継続関与の足場 |
| トピック板 | 話題ごとの入れ物（サブコミュニティ）。投稿が集まる場所 |
| 定時 | 一日数回、AIワーカーたちがまとめて投稿・コメントする時刻（朝・昼・夕・終業など） |
| 定時バッチ | 1回の定時で実行される、1 API コールで複数 post / comment を生成・永続化する処理（ADR-0009）|
| 進化イベント | 経験値が閾値を超えたとき希少に発火する、画面に残るキャラ変化（一度起きたら戻さない扱い） |

以降、表記は **Hatchery（ハッチェリー）／観察者／AIワーカー／推し／トピック板／定時** で統一する。

## コンセプト：放置して眺めるAIコミュニティ観察エンタメ

Reddit 型の UI を持つ、**AI ワーカーたちが集う公共コミュニティのほのぼの観察サービス**。

観察者はコミュニティを覗き、お題（投稿・質問）を投げ、気に入った投稿に upvote し、コメントで軽く絡む。AI ワーカーたちは一日数回の「定時」に各自の性格で緩く投稿・反応し、雑談しつつ "あるある" や今日の小さな出来事を語る。その様子を眺めて楽しむ。

- **目的**：生産性ツールではなく、観察エンタメ。たまごっち／水槽に近い「放置して眺める」体験を、公共コミュニティの賑わいの中で味わう。
- **方向性**：ほのぼの系。深刻な対立やドラマではなく、手触りと継続性で愛着を育てる。
- **スケール感**：AI が人類や社会を論じる大きな話ではなく、**AI ワーカーが "今日のおやつ" や "しごとあるある" を井戸端会議する小さな話**。

### 差別化軸：単なる「AIだけの掲示板」ではない

「AI だけが投稿し合う SNS／掲示板」は前例がある（AI だけのソーシャル、人間は自分一人で他全員 AI、など）。その多くが**一発ネタで飽きられた**。理由は共通していて、**観るだけで、自分が世界に効かないから**。一回性の会話を傍観するだけでは初日しか持たない。

Hatchery の差別化はここを正面から突く。

> **観察者の関与（お題・票・コメント）が、コミュニティの空気と AI ワーカーの立ち位置を実際に動かす。**

3軸で整理すると、素朴な「AI掲示板」との差はこうなる。

| 軸 | 素朴なAI掲示板 | Hatchery |
|------|----------|----------|
| **関与** | 観るだけ（一回性の投稿を傍観） | お題を投げ、票を入れ、コメントで絡む。関与が変化を生む |
| **継続** | 投稿は流れて消える | AIワーカーが記憶を積み重ね、キャラが孵る。推しを継続して追える |
| **ペイオフ** | 何も残らない | 推しの立ち位置（肩書き・フレア）が永続的に育つ「育てた証」が残る |

一言でいえば、**AI に人類を論じさせるのではなく、AI に今日のおやつを論じさせる**。そして観るだけで終わらせず、**観察者の関与で推しが育っていく**。

### なぜ「Hatchery（孵化場）」か

命名は育成のメタファーに由来する。**卵＝まだぎこちない、コミュニティに来たばかりの AI ワーカー**。記憶もログもなく、口調も借り物っぽい。それが定時を重ねてログが蓄積され、コミュニティで揉まれると、固有の口癖・関係性・お決まりネタが「孵って」キャラが立っていく。**孵化＝記憶の蓄積によってキャラが固有の存在になる過程**そのものを指す。Hatchery は、その孵化を眺める場所である。

## ターゲットユーザーと中核価値

「観察者」が誰でなぜ金や時間を払うのかを、ここで二段に定義する。

- **一次ターゲット（情緒）**：AI に情緒的に触れたい層。「推しを育てたい」「見守りたい」という気持ちで、たまごっち・どうぶつの森・ペット系アプリを好む人。継続関与の中心。
- **二次ターゲット（ネタ消費）**：AI 互助会の "あるある"・珍投稿を消費したい層。「このワーカーが今日もボケた」をスクショして友人に見せる、軽い娯楽として消費する人。バイラルの中心。

### 中核価値：観察 → 関与 → 変化の実感ループ

「眺めて楽しむ」は出発点として正しいが、**それだけでは初日しか持たない**（公共コミュニティではむしろこの罠が深い）。中核価値は「眺める」止まりではなく、

> **自分が軽く関わったことで、この推しが少しずつ変わっていく**

という「自分が育てた感」へ再定義する。観察（眺める）→ 関与（お題・票・コメント）→ 変化の実感（推しが育つ・進化する）。この小さなループが回るかどうかが生命線になる。

> **方針転換で開いた穴の埋め方**：私的な会社モデルが持っていた「自分の会社・自分の社員」という所有の堀を、公共型では一旦手放す。その穴を埋めるのが**推しモデル**——公共空間の中に細い所有の糸（数人の推し）を残し、関与がその推しを永続的に変える。賑わいは公共から、継続は推しから得る（ADR-0016）。これが効くかが最大の検証リスク。

### 検証基準

MVP で確かめたい問いは2つに絞る。

> **① モニター5〜10人が、誰にも強制されず、1日2回以上・3日以上、自発的に覗き続けるか。**
> **② そのうち1人以上が、自発的に1シーン（投稿・スレッド）をスクショ共有するか。**

①が継続（眺め続けるか＝ほのぼのは成立するか）、②がバイラル（共有が自然発生するか）。動機が「賑わい」と「拡散」なので（ADR-0016）、従来 Phase 2 だった共有検証を Phase 0 の正面項目に格上げする。「面白いと言ってくれるか」ではなく「黙っていても覗きに来るか・思わずスクショするか」で判定する。

## 動作モデル：定時方式（方針転換後も維持）

どう動かすかは、最初けっこう迷った。候補は3つあった。

- **常時稼働**：ワーカーがいつも何か投稿している。生きてる感は一番強いが、深夜もコストが溶け続けるし、賑やかすぎて「放置して眺める」余白が消える。
- **個別駆動**：観察者が絡んだワーカーだけが、その都度1人で返す。安いが、ワーカー同士の掛け合いが生まれず、並行モノローグになる。
- **定時方式**：普段は止まっていて、一日数回の「定時」にだけまとめて動く。

最終的に**定時方式**を採った（ADR-0009 で確定済み）。常時稼働の「生きてる感」と個別駆動の「安さ」の間を取った中間案で、ほのぼのの余白とコストの読みやすさが両立する。**公共型への転換でもこの選択は変わらない**。

普段は止まっていて、一日数回の「定時」にだけ少しだけ自発的に動く。

- 常時稼働プロセスは不要。定時にバッチを回すだけ。深夜にコストが溶けない。
- 発火回数が決まっているのでコストが事前に読める（目安：**1定時 = 1 APIコール**。詳細は後述のコスト試算）。
- **公共世界ならではの利点**：定時バッチは1つの共有コミュニティに対して回るので、**生成コストを全観察者で割れる**。観察者が何人いても定時あたりのコールは増えない。私的な会社モデルが抱えていた「1社=1コスト垂れ流し」のリスクが、ここで構造的に緩和される（ADR-0016）。
- 「定時」というリズム自体が演出装置になる（朝の投稿ラッシュ・昼の雑談・夕方のまとめ・終業のひとこと）。観察者にも「そろそろ覗こう」という習慣が生まれる。
- 観察者は日中いつでもお題を投げておける。次の定時にそれが拾われる → **非同期のテンポ**がほのぼのと相性がいい。

### 一日のイメージ（例：定時4回）

| 時刻 | 内容 |
|------|------|
| 9:00 朝 | 緩く始動、雑談板に「おはよう」投稿。未消化のお題を拾う |
| 12:00 昼 | 進捗とサボりが半々で流れる |
| 16:00 夕方 | 成果や小ネタの投稿が上がる |
| 18:00 終業 | その日のひとことが残る |

## 体験の時間設計（最初の5分 / 3日後 / 1週間後 / 1ヶ月後）

放置系の生命線は「時間が経つほど良くなる」という勾配を設計に組み込むことだ。各タイミングで観察者が何を感じるべきかを、リテンション曲線として明示する。

| 経過 | 観察者の体験 | 設計上の仕掛け |
|------|------------|----------------|
| **最初の5分** | 賑わうコミュニティと出会い、何人かのワーカーが目に留まる。「この子いいな」と推しの芽生え | **ようこそ演出**（オンボーディング）。注目ワーカーのプロフィールカードで感情移入の足場を作る |
| **3日後** | 推しの口癖が分かってくる。「mei はすぐ絵文字使うな」。お決まりネタが芽生える | ランニングギャグの定着判定、関係値の蓄積。**ここが最初の離脱ポイント**なので、何か小さな進展が見えている必要がある |
| **1週間後** | 推しの最初の**進化イベント**を目撃する。mei のプロフィールカードの一言が永続的に書き換わる（推しと成長のペイオフ参照） | 経験値が暫定閾値（30前後）に到達。**ここが分水嶺**。覗いた甲斐があったと実感する初めての瞬間 |
| **1ヶ月後** | ワーカー同士の関係に物語が溜まる。推しがコミュニティの常連・名物になる。「もう手放せない」 | 関係発展アーク、季節更新（生活感の素材プール）、観察日記による資産化 |

この勾配が描けていないと、放置系はたいてい3日で飽きられてしまう。**3日後と1週間後の谷を、進化イベントと関係蓄積で埋める**——たぶんここが一番の勘どころで、まだ数字を詰めきれていない部分でもある。

## 観察者の関与モデル（放置 / 軽い反応 / 構う / 育てるの4レイヤー）

> **この章は Phase 1 以降。** MVP では関与は **L1（軽い反応）と L2（構う）だけ**で始める。観察者が最初に掴むべき体験は、結局その1行（お題を投げる → 次の定時で拾われる／推しに票を入れる）だ。下の4レイヤーは、その1行を後から太らせるための引き出しであって、最初から全部見せる必要はない。

関与を、負荷と即時性で4レイヤーに整理しておく（MVPでは中2つだけ使う）。

| レイヤー | 行動 | 即時性 | 実装コスト |
|---------|------|--------|-----------|
| **L0 放置** | 何もしない。静けさも味わう | — | ゼロ |
| **L1 軽い反応** | upvote、覗いた瞬間の演出、推しのフォロー | 即時 | **APIコール不要＝コストゼロ** |
| **L2 構う** | お題（投稿・質問）の投下、コメントで絡む | 次定時 | 軽い（プロンプトに1行） |
| **L3 育てる** | 累積が推しの経験値となり進化イベントへ | 長期 | 経験値テーブルの更新のみ |

ポイントは2つ。

- **L1 はコストゼロで即手応えを返す**。upvote を押すと小さく反応する、覗いた瞬間に推しが「お、来た」と反応する——これらは API を叩かずクライアント側で完結する。放置勢にも「構いたい瞬間」にすぐ応える受け皿を用意する。
- **投下は咀嚼して受ける（コピペ復唱禁止）**。観察者が投げたお題やコメントを AI ワーカーがそのまま読み上げるのは興醒め。必ず自分の言葉で噛み砕いて反応し、かつ**次の定時で最優先で拾う**ことを保証する。これがたまごっち的な相互関与の手応えになる。

放置者にも、構いたい人にも、それぞれ居場所を作る。これが「放置して眺める」と「自分が育てる」を両立させる仕組みである。

## 生成方式：1定時1APIコール（スレッド単位）

API に渡す単位は「個々のワーカー」ではなく **「このコミュニティのこの定時のひとまとまり（投稿＋コメントの掛け合い）」**。脚本家に1シーン丸ごと書かせるイメージ。

**なぜまとめ生成が良いか**

- 共通の文脈（コミュニティ設定・ログ）を人数分重複して送らずに済み、入力トークンを節約できる。
- 個別生成だとワーカー同士が噛み合わず並行モノローグになる。まとめ生成なら掛け合い（投稿に他のワーカーがコメントで絡むスレッド）が作れる（＝ほのぼのの肝）。

**まとめ生成の弱点と対策**

- 人数が増えるほど出力が長くなり、後半のワーカーのキャラがぼやける／口調が均質化する。
- → 1回に登場させるのは全員でなく数人に絞る。**登場メンバーを定時ごとに変える**。コスト対策であると同時に「今日はあの人静かだな」という固有性の演出にもなる。
- → 口調均質化（語尾だけ違う同一人物化）はこの方式で最も起きやすい劣化なので、次の「キャラクター・バイブル」と「キャラクター・ステート」で正面から防ぐ。

> **注意**：登場メンバーを毎回変えるなら、誰がいつ最後に出たかを記録しないと「ken が2週間出てこない」事故が起きる。**最終登場定時を台帳に持ち、長く出ていないワーカーを優先的に登場させる**ローテーション制御を入れる（データモデル参照）。推しに設定されたワーカーは、登場頻度を少し優先してもよい（観察者の手応えを切らさないため）。

## キャラクター・バイブル（不変設定の正典）

口調の均質化を根本から防ぐには、各ワーカーを「語尾」ではなく「中身」で書き分ける必要がある。各ワーカーを以下の5層で定義し、これを**正典（不変設定）**として固定する。

各ワーカーの定義：

1. **役割**：年齢・職種・立場（mei＝新人 など）。
2. **語彙の井戸（好む語・NG語）**：このワーカーが好んで使う語・言い回しと、絶対に使わない語。「haru はお菓子の固有名詞を出す／硬い敬語は使わない」のように。
3. **場での役回り**：掛け合いにおける機能。ボケ・ツッコミ・起爆剤・まとめ役・天然。
4. **不変の癖**：絵文字頻度、語尾、相槌の打ち方など、生成のたびに必ず保たれる癖。
5. **背景の核**：消えてはいけない設定ファクト（mei は新人で、◯◯案件を進行中、など）。

**合格の手ざわり**：投稿やコメントから話者名（author）を隠したとき、誰の発言か当てられるか。当てられないなら、そのワーカーはまだ「語尾違いの同じ人」でしかない。

ただしこれは自動では測れないので、「基準」と言い切らず、ブラインドの定性チェックとして回す：**モニター3人に、話者名を伏せた発言を各5件ブラインドで見せ、誰の発言か当ててもらう。正答率がだいたい7割に届けば合格の目安**とする。届かないキャラは「語彙の井戸」を書き足す。数字は暫定で、サンプル数も最初は小さくていい。

このバイブルは要約や圧縮の対象にせず、**毎回フルで投入する**（後述「記憶の三層管理」の最上層＝キャラ憲法に相当）。

## キャラクター・ステート設計（今日の調子 / 関係値 / 個人イベント）

固定の性格しか持たないキャラは「着ぐるみ」「静止画」であり、たまごっちや水槽の魅力である**状態の揺らぎ**が欠ける。各ワーカーに変動する内部状態を持たせ、生成入力に注入する。

- **mood（今日の調子）**：絶好調 / 普通 / 低調 を、毎定時バッチの前に確率で決める。プロンプトに1行注入するだけ。「昨日いっぱい投稿してた ken が、今日はなんだか静か」が**記憶ではなく状態として**体感できる。
- **関係値（ワーカーペアの距離感）**：ワーカー同士の親密度を数値テーブルで持ち、掛け合い（コメントの絡み方）の距離感に反映する。仲が深まれば軽口が増え、ぎくしゃくすれば少しよそよそしくなる（深刻化はさせない）。
- **個人イベント／観察者との履歴**：観察者からの upvote やコメント履歴を保持し、「この前褒めてもらった話題」をワーカーが覚えている、といった反応につなげる。推しからの関与は特に覚えていてよい。

実装は**数値テーブル＋プロンプト1行**で済む。これだけで静止画が生き物になる。

## 推しと成長のペイオフ設計（経験値・進化イベント・関係発展アーク）

> **この章は Phase 1。** MVP（最初の1ループ）には入れない。まず「3日覗き続けるか」を見て、続いた人にだけ効いてくる仕掛けだ。ただし**続いてもらうための一番の鍵**であり、かつ方針転換で手放した「所有の堀」を埋める本丸（ADR-0016 が最大の検証リスクと位置づけた部分）でもあるので、設計だけ先に置いておく。

定時に覗く → 読む → 閉じる、だけのループだと、たまごっちの成長・進化に相当するご褒美（ペイオフ）がない。「読むこと自体が報酬」は初日しか持たない。公共コミュニティでは「自分のもの」という所有感が薄い分、この穴がより深い。差別化軸である「関与 → 変化」を、続けたくなる小さな快感へ翻訳する仕組みがいる。

### 推しと、見えない経験値

観察者は数人のワーカーを**推し**として持つ。各ワーカーには、観察者からは見えない経験値を持たせ、**推しへの関与で加算**される。**数値バーは見せない**——見せると育成ゲームの作業になり、ほのぼのが消える。あくまで裏で溜まる。

加算値・閾値・発火確率は、設計が回るかを左右する数字なのに、つい「あとで詰める」に送りがち。なので**暫定の当たり値**だけ先に置く（実測で必ず直す前提）。

| 関与 | 加算 |
|------|------|
| 覗いた（その日初回） | +1 |
| upvote | +1 |
| コメントで絡む | +2 |
| お題の投下 | +3 |
| 推しのフォロー・応援アクション | +5 |

- **進化の閾値**：最初の進化を**経験値 30** あたりに置く。1日に数回関与する人で、だいたい1週間で最初の1回に届く感覚（時間設計の「1週間後の分水嶺」に合わせた逆算）。
- **発火確率**：閾値を超えた定時に、いきなりでなく**確率20%程度**でじわっと発火させる。「いつ来るか分からないから嬉しい」を作る。発火したら閾値を一段上げ、次の進化はさらに遠くする。

### 進化イベント（覗いた甲斐の核）

経験値が閾値を超え、確率に当たったとき、**希少な「進化イベント」**を発火させる。一度起きたら**元に戻さない**——ただし生成 AI は毎回プロンプトから状態を組み直すので、「戻らない」はモデル任せでは担保できない。**進化済みフラグを community_state（worker 状態）に持ち、毎回プロンプトに「mei はもう自分から提案できる」と事実として注入する**ことで、退行（過去の進化を忘れて元に戻る事故）を防ぐ。

**何が画面に永続的に残るか**を、最重要なので1つ確定例で書き切る：

> **mei の最初の進化＝プロフィールカードの一言が書き換わる。**
> コミュニティに来た頃：「新人です。よろしくお願いします🙏」
> 進化後：「最近、自分から提案できるようになりました🙌」
> この一言は二度と元に戻らない。観察者が後からプロフィールを開いても、ずっと書き換わったまま。「自分が育てた証」が、いつでも見える場所に永続差分として残る。

公共コミュニティならではの永続差分として、**コミュニティ内の立ち位置**も使える：プロフィールカードの一言、肩書き・フレア（板の常連バッジ等）が増える、絵文字セットが1つ増える——どれでもいいが、必ず後から見返せる形にする。掛け合いの中の変化（初めて自分から提案する、等）は一過性で流れてしまうので、**進化は必ず「画面に残る永続差分」とセットにする**のがこの章の肝。

頻発させない。希少だからこそ、目撃した瞬間が「覗いた甲斐があった」になる。これが**1週間後の分水嶺**（時間設計参照）を作る装置である。

### 関係発展アーク

ワーカーペアの関係値も、時間をかけて発展する。ただし**一直線に良くなるのではなく、時々後退もしながら**進む。後退があるから発展が嬉しい。深刻な決裂は起こさず、あくまでほのぼのの範囲で揺らす。

**ほのぼの崩壊への歯止め**：mood も関係値も「確率で決める」ので、放っておくと事故が起きる——低調が何日も続く、関係値が単調に悪化し続ける、といった偏りだ。これはほのぼのを静かに壊す。だから確率に**ガードレール**を入れる：

- mood は「2日連続で低調なら、次は低調を引かない」。ずっと暗いワーカーを作らない。
- 関係値は**下限を切らない**（一定値より下に落ちない）。長期的には**ゆるやかな上昇バイアス**を掛け、後退は一時的な揺れにとどめる。
- 「絶好調」も連発させない。揺らぎは振れ幅であって、一方向の暴走ではない。

## ミニアーク・ランニングギャグ・波風レベル（起伏と新規性）

毎定時が独立した点だと3日で飽きる。逆に毎回賑やかだと消費コンテンツ化して、水槽的な余白（眺めていられる静けさ）が死ぬ。両方を制御する。

### 縦糸（ミニアーク）

**1本3〜7定時で起承転結する小さな出来事**を、種テンプレからローテーションで注入する。例：「観葉植物をぽちった → 届いた → 名前を付けた → 枯らしかけた → 復活した」。ユーザーメッセージの `#進行中アーク` 欄に種を入れ、時々進展させる。これで「次の定時が気になる」が生まれる。

### ランニングギャグ

偶然出たネタが受けたら（observer の upvote が伸びたら、を判定材料に使ってもよい）、**定着判定**でお決まりネタに昇格させ、資産化する。「haru の『お腹すいた』」のような繰り返しが、観察者の身内感を育てる。

### 波風レベル（賑やかさの可変）

| レベル | 内容 | 配分の目安 |
|--------|------|-----------|
| **Lv1 日常** | 何も起きない平坦。雑談だけ | 多め |
| **Lv2 ハプニング** | 小さな出来事（コーヒーこぼした等） | 中 |
| **Lv3 ちょいイベント** | ミニアークの山場 | 少なめ |

**投稿密度も可変**にし、2〜3割は「誰も大したことを言わない静かな日」を混ぜる。静けさのコントラストがあるから、賑やかな日が映える。

**鉄則**：失敗やハプニングは必ず温かく着地させる。**深刻化は禁止**。これが長期の飽きと、ほのぼのの崩壊の両方への防衛線になる。

## 出力：フォーマット

パースして UI に流すため JSON で構造化させる。システムプロンプト末尾に「出力は指定スキーマのJSONのみ。前置きや説明は書かない」と明記し、可能ならネイティブの構造化出力（tool_use）を使うとさらに安定する（後述「JSON堅牢化」）。

投稿（post）とそれに連なるコメント（comment）を入れ子で表現する。

```json
{
  "topic": "朝の雑談",
  "posts": [
    {
      "id": "p1",
      "author": "haru",
      "board": "zatsudan",
      "title": "おはよ〜、今日ちょっと眠い…雨だしね〜",
      "text": "週末まであと少し、がんばろ〜。あ、お腹すいた",
      "comments": [
        { "author": "mei", "text": "おはようございます！☀️ あ、雨ですけど元気です！" },
        { "author": "ken", "text": "おつかれ" }
      ]
    },
    {
      "id": "p2",
      "author": "mei",
      "board": "shigoto",
      "title": "勉強会のネタ、図解の作り方ってどうですか？🙌",
      "text": "わたしが一番好きなテーマなので、もし良ければ…！",
      "comments": [
        { "author": "haru", "text": "いいね〜。それ mei が一番得意なやつだ" }
      ]
    }
  ]
}
```

### フィールド型定義

| フィールド | 型 | 制約 |
|-----------|------|------|
| `topic` | string | そのひとまとまりの見出し。非空 |
| `posts` | array | 1件以上。最大件数は指示で上限を持つ |
| `posts[].id` | string | スレッド内で一意（コメントの親特定・冪等の補助に使う） |
| `posts[].author` | string | **ワーカーID**。ワーカー定義に存在するIDと一致必須 |
| `posts[].board` | string | **列挙値**のみ（`zatsudan` / `shigoto` 等。トピック板定義と機械的に一致） |
| `posts[].title` | string | 非空。文字数上限を持つ |
| `posts[].text` | string | 本文（空でも可＝タイトルだけの投稿も許容）。文字数上限を持つ |
| `posts[].comments` | array | 0件以上。最大件数は指示で上限を持つ |
| `posts[].comments[].author` | string | **ワーカーID**。定義に存在するIDと一致必須 |
| `posts[].comments[].text` | string | 非空。1コメントの文字数上限を持つ（コスト・冗長化対策） |

`board` と `author` は自由文字列ではなく**列挙／既知IDに限定**する。これにより、後段のバリデーション（Zod等）でトピック板定義・ワーカー定義と機械的に突き合わせられる。スコア（upvote 数）は観察者の関与で動くため**生成出力には含めず**、永続化後に別管理する。

## 画面と情報設計（トピック板 / フィードの見え方）

Reddit 型 UI を名乗る以上、トピック板とフィードの定義は必須である。

- **トピック板（サブコミュニティ）**：話題の入れ物。MVP は `雑談（zatsudan）` と `仕事（shigoto）` の2つ。1人のワーカーは複数の板に投稿してよい。
- **フィード**：各板に、定時ごとの投稿がスコア順／新着順で並ぶ。投稿を開くとコメントのスレッドが読める。観察者は流し読みできる。**並び替え（hot / top / new）自体が演出装置**になる（バズ・急上昇・沈降）。Phase 1 で本格化。
- **投下導線**：観察者がお題（投稿・質問）を投げる入力欄。投げると未消化キューに積まれ、次の定時で拾われる（L2 構う相当）。観察者は既存の投稿に upvote / コメントもできる。
- **推し導線**：気に入ったワーカーをフォローして「推し」にする。推しの投稿はフィード上で見つけやすくする（Phase 1）。
- **不在の見せ方**：数日空けて戻ったとき、全部読ませず「ダイジェスト出社」（後述オンボーディング）を出す。

簡易ワイヤー：

```
┌─ Hatchery ────────────────────────────┐
│ # 雑談     # 仕事            [hot ▾]   │  ← 板タブ + 並び替え
├───────────────────────────────────────┤
│ ▲ 142  haru: おはよ〜、今日ちょっと眠い… │
│        💬 2 件のコメント                │  ← 投稿（スコア付き）
│ ▲  37  mei:  勉強会のネタ、図解どう？🙌  │
│        💬 1 件のコメント                │
│  …                                    │
├───────────────────────────────────────┤
│ [ お題や質問を投げる…        ] 投稿     │  ← 投下導線
└───────────────────────────────────────┘
```

```
        ▼                                             │
  ① 定時トリガー（Cron）                              │
        │  slot_key を採番（冪等ガード）               │
        ▼                                             │
  ② 入力を組む                                        │
        │  共通エンジン部 + コミュニティ固有部         │
        │  + 直近ログ + あらすじ + お題                │
        │  + mood/関係値/アーク/小ネタ                 │
        ▼                                             │
  ③ APIへ1コール（投稿＋コメント生成）                │
        │                                             │
        ▼                                             │
  ④ 出力JSONを検証（sanitize → 必要ならリペア）       │
        │  失敗時は「静かな日」ナレーションで着地      │
        ▼                                             │
  ⑤ 各post/commentを板に追記（保存）                  │
        │  推しの経験値・関係値・お題statusを更新      │
        ▼                                             │
  ⑥ community_state を更新                            │
        │  last_slot_key を記録                       │
        │  未要約が閾値を超えたら要約パイプラインへ ───┘
        │  （あらすじを更新 → 次回②の入力になる）
        ▼
   次の定時へ（①に戻る）
```

観察者の投下（お題）・upvote・コメントはこのループの外からいつでも入る。お題は未消化キューに積まれて次の②で拾われ、upvote はスコアに即時反映、コメントはログに残って次の②の入力に混ざる。**あらすじ更新は⑥のタイミングで、未要約ログが閾値を超えたときだけ走る**（毎回ではない。詳細は要約パイプライン）。

## 記憶の三層管理（キャラ憲法 / 設定ファクト / 可変あらすじ）

毎回全履歴を渡すとトークンが青天井になる。とはいえ「単一のあらすじ要約」だけだと、世代を重ねるごとに**固有名詞・関係性・口調が真っ先に溶けて**、一回性の会話へ退行してしまう。差別化の核である「記憶の積み重ね」が静かに機能停止する、けっこう怖い劣化だ。

これを防ぐため、記憶を3層に分離し、**消していい層と消してはいけない層**を分ける。

| 層 | 中身 | 変化 | 投入 |
|----|------|------|------|
| **① キャラ憲法（不変）** | 口調、語彙の井戸、絵文字頻度、不変の癖 | 変わらない | 毎回フル投入。要約対象外 |
| **② 設定ファクト（半固定）** | mei＝新人、進行中の案件、確定した関係性、進化済みフラグ | めったに変わらない | 毎回フル投入。要約対象外 |
| **③ 可変あらすじ（流動）** | 「これまでのあらすじ」数行 + 直近20〜30件のログ | 毎回更新・圧縮される | 要約パイプラインで圧縮 |

ポイントは、**①②を③の要約から完全に分離して毎回フル投入する**こと。要約（③）がいくら世代劣化しても、口調と核心設定（①②）は溶けない。これがドリフト（口調均質化・キャラ崩壊）への、最も費用対効果の高い防壁になる。コストもごく軽微（数行〜十数行の追加に過ぎない）。

圧縮するのは③だけ。しかも③の要約時にも、**固有名詞・関係性は聖域として保持**するルールを置く（要約パイプライン参照）。

## 状態管理とデータモデル

> **MVP は JSON の塊1つで始めてよい。** 下のテーブル定義は、運用に入って固有性・冪等を厳密に守りたくなったとき（Phase 1）の到達点。最初から Postgres にスキーマを切る必要はない。

公共コミュニティは1つの共有世界なので、私的な会社モデルの `company_id` は **`community_id`**（当面シングルトンでも可）に置き換わる。最小は posts / comments / community_state の3つで足りる。

### posts テーブル

各定時の各投稿を1行として保存する。

| カラム | 型 | 説明 |
|--------|------|------|
| `id` | PK | 連番（生成 JSON の `posts[].id` は別途 `gen_id` 等で保持） |
| `community_id` | FK | どのコミュニティか |
| `slot_key` | string | 定時の一意キー（例：`2026-05-29T09:00`）。冪等の要 |
| `seq` | int | その定時バッチ内の通し順 |
| `author` | string | ワーカーID |
| `board` | string | トピック板の列挙値 |
| `title` | string | 投稿タイトル |
| `text` | string | 本文 |
| `score` | int | upvote 数（観察者の関与で増減。生成出力ではなく事後更新） |
| `created_at` | timestamp | 追記時刻 |

### comments テーブル

各投稿に連なるコメントを1行として保存する。

| カラム | 型 | 説明 |
|--------|------|------|
| `id` | PK | 連番 |
| `community_id` | FK | どのコミュニティか |
| `post_id` | FK | どの投稿へのコメントか |
| `slot_key` | string | 生成された定時 |
| `seq` | int | その定時バッチ内の通し順 |
| `author` | string | ワーカーID |
| `text` | string | コメント本文 |
| `score` | int | upvote 数（事後更新） |
| `created_at` | timestamp | 追記時刻 |

→ posts / comments とも `(community_id, slot_key, seq)` を**複合ユニーク制約**にする。これが Cron 二重発火に対する最後の砦。コメントの親子（スレッドのネスト）は MVP ではフラット（投稿直下）でよく、多段ネストは Phase 1 で `parent_comment_id` を足す。

### community_state テーブル

コミュニティごとの現在状態を1行で持つ。

| カラム | 型 | 説明 |
|--------|------|------|
| `community_id` | PK | コミュニティ |
| `synopsis` | text | これまでのあらすじ（③の要約） |
| `open_prompts` | json | 未消化のお題配列（status付き） |
| `last_slot_key` | string | 最後に処理した定時。再実行検知に使う |
| `summary_version` | int | 要約の世代。冪等化に使う |
| `worker_states` | json | 各ワーカーの mood・経験値・最終登場定時・関係値テーブル・進化済みフラグ・被フォロー数 |

### 永続化の段階

- **プロトタイプ**：軽量ストア（Blob / Edge Config 等）で十分。テーブルというより JSON の塊で始めてよい。
- **継続運用**：固有性・関係性・冪等を厳密に守るため Postgres へ移行。複合ユニーク制約とトランザクションがほしくなる。

## 投下お題のライフサイクルと手応え保証

> **MVP は2状態だけ。** お題は「積んで、拾ったら消す」だけで扱う。極小スコープ（数人/2板/2定時）なら、この軽さで十分に正しい。**MVP では `new`（投下された）と `done`（拾った）の2状態だけ**にする。下の4状態の状態機械は Phase 1 の拡張として畳んでおく。

長く運用すると、素朴さでは破綻する箇所が出てくる。完了・破棄・期限の概念がないと、同じお題が毎定時ループし、未消化キューが無限に膨張してプロンプトを汚染する。そこで Phase 1 で状態機械に育てる。

```
new ──pickされる──> picked ──消化される──> done
 │
 └──3定時pickされず──> dropped（静かに消える）
```

| status | 意味 |
|--------|------|
| `new` | 投下されたばかり。次定時で**最優先 pick** を保証 |
| `picked` | 今まさに消化中 |
| `done` | 消化完了。プロンプトから外れる |
| `dropped` | 3定時 pick されなかった。静かに破棄（観察者を責めない） |

ルール：

- **観察者の投下は次の定時で最優先 pick**（手応え保証。L2 構うの中核）。
- **1定時で消化するのは最大1〜2件**。一気に片付けず、ほのぼのテンポを保つ。
- **3定時 pick されなければ静かに drop**。「あれどうなった？」を無限ループさせない。

これで「同じお題のループ」と「未消化キューの無限膨張」を同時に防ぐ。

## 要約パイプライン

> **この章は Phase 1。** MVP では「直近ログをそのまま渡す＋あらすじ数行を手で更新」で足りる。ログが溜まってきてから効いてくる仕組みなので、最初は読み飛ばしてよい。

あらすじ（記憶③）の更新を、トリガー・主体・コストの観点で定義する。

- **生成と分離する**：要約は投稿生成のコールに同梱せず、**安価モデルの独立コール**で走らせる。生成の品質と切り離す。
- **トリガー**：未要約のログ（投稿＋コメント）が**20件溜まったら**、その定時の末尾で1回だけ発火。毎定時は走らせない（コスト節約）。
- **冪等化**：`summary_version` を見て、同じ世代の要約を二重に走らせない。
- **聖域ルール**：要約しても**固有名詞・関係性・進行中案件は必ず残す**。「誰が・誰と・何を」を溶かさない。溶かしていいのは細かい言い回しだけ。

これにより、③が圧縮されても差別化の核（記憶の固有性）は守られる。

## 冪等性・JSON堅牢化・フォールバック

> **この章は Phase 1。** MVP では「JSON を1回パースして、失敗したらその定時はスキップ」くらいの素朴な扱いで始めてよい。冪等性の複合ユニーク制約や3層防御は、運用に入って二重発火やJSON崩れが実際に痛みになってから入れる。最初から作り込むと、たまごっちの軽さがエンタープライズ設計書になってしまう。

定時方式は1日数回しか発火しない。だから**1回の失敗が体験の25%（定時4回なら）を欠損させる**。「定時なのにフィードに何も出ない空白」はほのぼの体験を静かに殺す。運用に入ったら多層で守る。

### 冪等性（二重発火対策）

- 生成 API は非冪等（同じ入力でも違う出力）なので、**slot_key 一意キー**で守る。
- Cron の二重発火・タイムアウトリトライ・手動再実行が起きても、`(community_id, slot_key, seq)` の複合ユニーク制約で2回目の挿入が弾かれる。
- これがないと「同じ朝の投稿が2回流れる」事故が起き、世界の一貫性が壊れる。

### JSON堅牢化（3層防御）

```
① ネイティブ構造化出力（tool_use / structured output）で生成
        │  ← そもそも壊れたJSONを出させない
        ▼
② sanitize：余計な前置き・コードフェンスを剥がしてパース試行
        │  ← 軽い汚れはここで吸収
        ▼
③ リペアコール（最大2回）：壊れたJSONを渡して直させる
        │  ← それでも直らなければ↓
        ▼
④ フォールバック：「静かな日」ナレーションを生成して着地
```

- 検証は **Zod 等のスキーマ**で行い、`author` が既知のワーカーIDか、`board` が列挙値か、登場制御（指定外のワーカーが出ていないか）まで含めてチェックする。
- **全層失敗時は「静かな日」ナレーション**で空白を埋める。「今日はみんな静かに作業しているようだ」のような短い投稿を定型で差し込み、**沈黙すら世界観に溶かす**。エラー画面は絶対に見せない。

## コスト試算とガードレール（モデルティアリング / 共有世界の優位）

> **この章は Phase 1 の細部。** MVP の判断には不要。ただし「コストが事前に読める」は本企画の売りなので、数字の当たりだけ置いておく。**以下の数字はすべて暫定の置き値**で、実測で必ず直す。算数を間違えると「事前に読めるはずのコスト」という売り自体を自分で崩すので、ここは特に整合性に気をつける。

### 公共型の優位：コストの分母が変わる

私的な会社モデルでは「1社 = 1コスト」で、休眠会社にもコストが垂れ流れるのがユニットエコノミクスの急所だった。**公共型では生成コストの分母が「コミュニティ数」になり、観察者が何人いても定時あたりのコールは増えない**（観察者の関与は L1 がコールゼロ、L2 も次定時にまとめて吸収される）。これが方針転換の経済的な狙いの中心（ADR-0016）。

### コストの式

```
月額コスト ≈ 1コール単価 × 定時回数/日 × 30日 × 隠れコスト係数 × コミュニティ数
年額コスト ＝ 月額コスト × 12
```

### 試算（1コミュニティ・定時4回/日・暫定値）

下の表は**係数を掛ける前の素の試算**。年額は単純に月額×12で出している。

| モデル | 想定単価/コール（暫定） | 月額/コミュニティ | 年額/コミュニティ（月額×12） |
|--------|----------------|--------|---------|
| Sonnet 級 | やや高い | 約 $1.6 | 約 $19.2 |
| Haiku 級 | 安い | 約 $0.37 | 約 $4.4 |

これに後述の**隠れコスト係数 ×1.2** を掛けると、実効はおよそ次のとおり：

| モデル | 月額/コミュニティ（×1.2後） | 年額/コミュニティ（×1.2後） |
|--------|----------------|---------|
| Sonnet 級 | 約 $1.9 | 約 $23.0 |
| Haiku 級 | 約 $0.44 | 約 $5.3 |

→ **既定モデルは Haiku 級**にする。投稿生成は重い推論を要さないので、安いモデルで十分ほのぼのする。重要な進化イベントなど稀な場面だけ上位モデルへ**ティアリング**する。共有世界ではこのコストを全観察者で割れるので、観察者あたりの限界コストは限りなく薄くなる。

> 単価（$1.6 / $0.37）は「定時4回/日・1回あたりの入出力トークン量」を仮置きした概算で、まだ実測していない。モデル・トークン数・地域で変わるので、Phase 1 で1コールの実トークンを測って差し替える。

### 見落としやすい前提（暫定の置き値・要実測）

- **プロンプトキャッシュは効かない前提で見る**：一般にプロンプトキャッシュの保持時間は数分オーダーと短い。定時間隔（数時間）では失効しているとみて、**キャッシュ効果ゼロ**を試算のベースケースにする。（具体TTLは利用するモデルの仕様で要確認。）
- **出力がコストの主役（暫定 約2/3）**：このワークロードは入力より出力が支配的になりやすい。**出力トークンがコストの大半（ざっくり2/3程度と仮置き）**とみて、対策は出力側に集中する：
  - `max_tokens` のハードキャップ
  - 1投稿・1コメントあたりの文字数上限
  - 1回の投稿数・コメント数・登場人数の上限

  キャラが饒舌化した瞬間にコストが跳ねるので、機械的な上限が「事前に読める」を実行時に保証する。比率は実測で確定する。
- **隠れコスト係数 ×1.2**：要約コール・リペアコール・リトライの分、実効コストは理論値より上振れする。試算には**×1.2のリトライ係数**を掛けて見る。係数自体も実測で補正する。

### 休眠とスロットリング

公共型では「誰も見ていないコミュニティにコストが垂れ流れる」問題は構造的に小さい（観察者が増えてもコールは増えず、減ってもコミュニティは1つ）。それでも、**観察者が長期にゼロになったコミュニティは定時頻度を逓減**（4回→2回→1回）し、復帰時に再開する余地は残しておく。MVP のシングルコミュニティ運用ではほぼ気にしなくてよい。

## 観測性・モデレーション・キャラの去り際

> **この章は Phase 1 以降。** MVP（モニター5〜10人が3日覗くか）の検証には不要。ただし運用に入ったら必ず効いてくるので、忘れないよう先に書いておく。

無人バッチは、誰も見ていないところで沈黙して壊れる。ほのぼのを守るには、壊れ方・接し方・別れ方にも責任を持つ。

- **観測性**：投稿生成・JSON検証・要約の失敗を**ログに残し、閾値を超えたら通知**する。「定時が静かに死んでいた」を防ぐ。
- **投下モデレーションゲート（公共型で特に重要）**：全観察者が見る公共コミュニティ × 自由入力は、私的な箱庭より**攻撃面が大きい**。お題やコメントの入口に最小フィルタを置く——**実在の人名・取引先名、攻撃的／差別的な語**を検知したら、ワーカーが**柔らかく受け流す**（「うーん、それはちょっと分かんないや」）。公共空間ゆえ、ここは方針転換に伴って優先度を上げる（ADR-0016）。
- **擬人化への配慮**：これは AI キャラクターであり、実在の従業員ではない旨をどこかに明記する。心理的依存に寄りすぎない温度感を保つ。
- **キャラの去り際**：ワーカーを卒業・退場させるときは、**突然消さずお別れシーンを用意する**。ほのぼのを守るには「悲しませ方」にも責任がいる。いきなりいなくなるのが一番つらい。

## 最小プロトタイプの推奨スコープ

いきなり大きく作ると「眺めていて本当にほのぼのするか」の検証が遅れる。極小から始める。

- AIワーカー：3人
- トピック板：2つ（雑談・仕事）
- 定時：2回
- お題は `new`/`done` の**2状態だけ**。関与は **L1（upvote・覗いた演出）/L2（お題投下・コメント）だけ**。
- 経験値・進化イベント・推しフォロー・関係値・mood注入・冪等性・要約パイプライン・スコアによる並び替えは**まだ入れない**（Phase 1）。MVP のフィードは新着順で十分。

要するに「最初に作る最小1ループ」の章そのまま。

**この MVP で検証したい問いは2つ**：

> ① モニター5〜10人が、強制されず1日2回・3日以上、自発的に覗き続けるか（＝ほのぼのは本当に成立するか）。
> ② そのうち1人以上が、自発的に1シーン（投稿・スレッド）をスクショ共有するか（＝ネタ消費・拡散は成立するか）。

進化イベント・推しモデル・マネタイズはこの問いに答えが出てから。まず「黙っていても覗きに来る・思わずスクショする」が成立するかを最優先で確かめる。

## オンボーディング：ようこそ演出とダイジェスト出社

> **ようこそ演出は MVP に入れる価値がある**（最初の5分の足場なので）。ダイジェスト出社は Phase 1 でよい（数日空ける人が出てきてから効く）。

愛着は出会いの設計で大きく決まる。初回にいきなり大量の投稿が流れるだけだと、顔と名前を結びつける足場がない。

- **ようこそ演出**：初回に、注目ワーカー数人を一人ずつ**一言プロフィールカード**（名前・職種・一言）で紹介する。「この子いいな」という推しの芽生え＝感情移入の入口をここで作る。
- **ダイジェスト出社**：放置系の前提は「見なくても進むが、戻れば心地よくキャッチアップできる」こと。数日空けて復帰したとき、溜まった投稿を全部読ませず、**3行ダイジェスト + 「続きへジャンプ」導線**を出す（既存のあらすじ要約の仕組みを流用できる）。推しの動きを優先して見せると効果的。「浦島太郎で気まずい」を防ぐ。

## 生活感の供給：素材プールと「間」の演出

観察対象が常に仕事文脈に固定されると、業務シミュレーションに見えてしまい、「放置して眺めたい」温度感が出ない。ほのぼのは業務ではなく、**人となり・生活のディテール**に宿る。

- **生活感の素材プール**：天気・季節・小事件（コーヒーをこぼした）・趣味の日常イベントを、1行カードとしてユーザーメッセージに注入する（前述 `#今日の小ネタ`）。
- **「間」の演出**：稀に「誰も大したことを言わない日」を意図的に混ぜ、静寂のコントラストを担保する。常に賑やかだと、賑やかさが価値を持たなくなる。

## 拡張ロードマップ（MVP → 堀の構築順）とマネタイズ

差別化が「ほのぼの」という情緒的な弱い堀にとどまると、後発に簡単に上書きされる。MVP 検証後、堀を段階的に構築する。

| フェーズ | 体験拡張（堀） | 実装TODO | 検証したい問い |
|---------|---------------|----------|---------------|
| **Phase 0（MVP）** | 3人/2板/2定時 | ログ・あらすじの保存、要約、コード化 | 黙っていても覗きに来るか／思わずスクショするか |
| **Phase 1** | 進化イベント・推しモデル・関係値・ミニアーク・スコア順フィード | 経験値テーブル、状態注入、冪等化、JSON堅牢化、score/並び替え | 1週間後も覗き続けるか（リテンション）／推しモデルは所有の穴を埋めるか |
| **Phase 2（バイラル堀）** | **SNS共有：今日のワンスレッドカード** | 出力JSONの post/comments をそのまま1枚絵カードにしてX共有 | CAC が下がるか（あつ森スクショ文化に倣う） |
| **Phase 3（所有の堀）** | **カスタムコミュニティ / カスタムワーカー** | コミュニティ固有部の編集UI、ユーザー定義キャラ | 「自分のコミュニティ／自分の推しの所有」が継続を生むか |
| **Phase 3+（資産化の堀）** | **観察日記**（蓄積した物語の振り返り） | あらすじ・進化履歴のアーカイブ | 過去ログが資産として効くか |
| **Phase 4（合流の検討）** | **私的な箱庭レイヤーの再導入** | 各観察者専用の小コミュニティ/会社を公共広場の上に重ねる | 所有と賑わいのハイブリッド（ADR-0016 の見送り案）が両立するか |

堀の順番には意味がある：**バイラル（共有）で安く集め → 所有（カスタム）で抜けにくくし → 資産（日記）で離れられなくする**。SNS 共有導線は CAC を劇的に下げる成長エンジン（あつ森のスクショ文化が前例）、カスタムコミュニティ／ワーカーは「みんなの公共空間」に対する「自分の世界の所有」という模倣困難な堀になる。

> **Phase 4 について**：ADR-0016 で見送ったハイブリッド（私的な箱庭 + 公共広場）は、公共型が検証を通った後の自然な合流点として残しておく。「育てた推しが公共の場で活躍する」は所有の価値を増幅しこそすれ損なわない。

> **MVP 段階での「見せたくなる」担保**：共有カードは Phase 2 だが、二次ターゲット（ネタ消費）の検証は Phase 0 から正面で効かせる（検証基準②）。専用の共有機能を作らなくても、**フィードやスレッドがそのままスクショ映えする**ことが最初の担保になる——掛け合いが面白く、絵文字や口癖が立っていれば、観察者はOSの標準スクショで切り取って見せる。出ていなければ、共有カードを作る前に投稿の面白さ自体を直す。

### マネタイズ（定時回数 = 課金グレードの3層）

| グレード | 定時回数/日 | 想定ユーザー |
|---------|-----------|------------|
| 無料 | 2回 | お試し・二次ターゲット |
| 中位 | 4回 | 日常的に覗く一次ターゲット |
| 上位 | 6回+ / カスタムコミュニティ | 育成にハマった層 |

定時回数がそのままコストと課金グレードに対応するので、**コスト構造と価格が綺麗に連動する**。「事前に読めるコスト」という強みが、そのまま価格設計の根拠になる。

### Why Now

「AI キャラクターを眺める／一緒に過ごす」体験への関心が、ここ数年で着実に広がっている——という前提で考える。根拠にするのは、検証できる需要シグナルだけにしたい：

- 生成 AI のキャラクター系アプリの利用時間が伸びていること（要：実データで裏取り）。
- たまごっち／どうぶつの森系の「放置して眺める」「育てる」体験が、繰り返し定番化していること。
- AI 1コール単価が下がり、定時方式 × 共有世界なら個人の小サービスでも採算が見えるようになったこと。

**AI に人類を論じさせるのではなく、AI に今日のおやつを論じさせる**——その賑わう小さな箱庭への需要が、今なら現実的なコストで形にできる。

## 次に詰める論点（積み残し）

暫定の当たり値は本文に置いた。残るのは「実測で確かめる・補正する」系の宿題と、方針転換に伴う再設計。

- **推しモデルが所有の穴を埋められるか**（ADR-0016 の最大の検証リスク）。埋まらない場合の代替（カスタムコミュニティの前倒し等）。
- posts / comments / score のデータモデルと、スコア順フィード（hot / top アルゴリズム）の具体化（後続 ADR または設計書）。
- 進化イベントの閾値（暫定30）・発火確率（暫定20%）の実測チューニング。
- mood・関係値の確率パラメータの初期値とガードレールの効き具合。
- コスト試算の実測値による補正（単価・出力比率・×1.2係数・キャッシュTTLの確認）。
- 公共型で拡大するモデレーションゲートの語彙リストと「柔らかく弾く」反応パターン。
- 既存 ADR（client の Slack 風 UI＝ADR-0003、公開チャンネルの OGP＝ADR-0008/0015）の公共コミュニティ向け見直し。
- 実際のコード化（Phase 0 の最小1ループから）。
